

## 目的路线

你现在迷失是正常的，因为这个项目已经从“能跑起来”进入了更难的阶段：**从功能正确，转向性能、稳定性、工程化闭环**。这两个阶段的判断标准完全不同。现在的 benchmark 结果不能说“烂”，但也绝对不能说“已经像个高性能 KV”。更准确的结论是：

**当前项目进度：功能骨架基本成型，单机/小规模 Raft 写链路能工作，观测体系刚开始可用；性能上还处在初版验证阶段，远没到优化完成。**

置信度：高。

**现在项目到底做到哪了**
已经完成或接近完成的部分：

1. **基础 KV 写链路能跑**
   
   - SDK -> route -> storage -> replica -> raft -> WAL -> apply，这条链路已经贯通。
   - 三副本 Raft 写入能提交。
   - benchmark 能压出稳定数据。

2. **Raft 基本语义有测试覆盖**
   
   - leader election
   - log replication
   - leader crash
   - minority cannot commit
   - snapshot catchup
   - recovering follower/leader
   - read index current term check

3. **持久化和恢复已经有雏形**
   
   - WAL append
   - snapshot
   - crash recovery
   - corrupted WAL repair/cut tail

4. **可观测性刚刚建立**
   
   - latency histogram
   - counter
   - Prometheus scrape
   - Grafana dashboard
   - p95/p99/avg 已经能看

这些说明项目已经不是玩具 demo 的最早期了。

但还没到“系统靠谱”的阶段。

**benchmark 结果到底好不好**
目前这组 benchmark：

- QPS：约 `300/s`
- avg：约 `3.5ms`
- p95：约 `5.3ms`
- p99：约 `9.7ms`
- failure：极低，约 `0.037/s`
- 主要失败：`NOT_YET_COMMIT`
- 主要尾延迟来源：leader 同步 AppendEntries RPC / `flush_messages`

这个结果的真实含义是：

**低并发/低 QPS 下延迟还可以，但吞吐没有证明出来，尾延迟已经暴露出 Raft 同步复制实现比较粗。**

不是灾难。  
但也不是优秀。

如果你是做一个教学型 Raft KV，这结果已经不错。  
如果你想做一个“像样的分布式 KV 原型”，现在还只是第一阶段完成。

**现在最大的问题不是 QPS 低，而是 benchmark 还没有把系统压明白**
300 QPS 这个数看起来像是 benchmark 目标速率，而不是系统极限。也就是说现在还不知道：

- 系统最大 QPS 是多少
- 从多少 QPS 开始 p99 爆炸
- CPU 有没有打满
- leader 是不是单线程卡住
- gRPC 是否成为瓶颈
- WAL fsync 是否在高压下变成瓶颈
- follower 落后时是否导致大量 `NOT_YET_COMMIT`

所以现在不要急着说“性能不好”。更准确地说：

**当前 benchmark 只能证明系统在 300 QPS 下基本稳定，不能证明系统性能上限。**

**我认为下一阶段方向应该这样排**
别同时抓一堆东西。按优先级来。

**阶段 1：把 benchmark 做成可信工具**
目标：能回答“系统在什么压力下坏掉”。

你现在应该补：

1. 阶梯压测
   
   - 100 QPS
   - 300 QPS
   - 600 QPS
   - 1000 QPS
   - 1500 QPS
   - 2000 QPS
   - 直到 p99 明显恶化或错误率上升

2. 每轮记录：
   
   - attempted QPS
   - success QPS
   - failure QPS
   - avg/p95/p99
   - not_leader
   - not_yet_commit
   - append_entries_rpc latency
   - WAL latency
   - apply latency

3. 输出一张表，而不是只看 Grafana 图。

这个阶段完成后，你才知道系统瓶颈在哪里。

**阶段 2：优化当前已经暴露的写路径问题**
现在证据最强的瓶颈是：

```text
Replica::put()
  -> raft_node->propose()
  -> flush_messages()
      -> 串行同步 AppendEntries RPC
  -> apply_committed_entries()
  -> 检查 commit_index
```

这里的问题是：

1. leader 给两个 follower 发 AppendEntries，当前是同步串行。
2. 写请求会被 RPC flush 阶段拖住。
3. 一轮 flush 不一定提交，导致偶发 `NOT_YET_COMMIT`。

优先优化：

- AppendEntries 并发发送
- 等多数派即可，不必等所有 follower
- 对 `NOT_YET_COMMIT` 做短暂 wait/retry，而不是立刻返回
- 区分“本次请求未提交”和“后台稍后会提交”的语义

这是当前最值得做的性能优化。

**阶段 3：扩大系统能力**
等单 shard 写链路稳定后，再做：

1. 多 shard benchmark
2. 多 table benchmark
3. read benchmark
4. mixed read/write benchmark
5. follower lag benchmark
6. leader failover during workload
7. snapshot during workload

不要现在就做太多复杂场景。当前单 shard 写链路还没压清楚。

**阶段 4：工程化稳定性**
之后才是：

- 配置管理
- 日志分级
- metrics 命名收敛
- dashboard 整理
- e2e 自动化
- chaos test
- 文档化 benchmark 结果

**你现在最应该做什么**
我建议下一步非常具体：

1. 先不要改 Raft 算法。
2. 先写一个阶梯 benchmark 脚本。
3. 每档 QPS 跑 60-120 秒。
4. 自动采集 Prometheus 指标。
5. 生成一份 markdown 表格。
6. 找到 p99 和错误率的拐点。
7. 再决定优化 `flush_messages`。

如果现在直接优化，很容易陷入“凭感觉改代码”。你现在缺的不是努力，是一张清楚的性能地图。

**目前项目状态一句话**
这个项目现在处于：

```text
功能正确性初步完成
        ↓
观测与 benchmark 初步完成
        ↓
正在进入性能瓶颈定位阶段
        ↓
还没有进入系统性性能优化完成阶段
```

所以你不应该沮丧。  
但也不应该自我感觉良好。  
现在最理性的判断是：**项目已经有了可优化的基础，但还没有足够的 benchmark 证据支撑“性能好”或者“性能差”的最终结论。**

下一步我建议做：**阶梯压测 + 自动汇总报告**。这会立刻把方向拉清楚。
