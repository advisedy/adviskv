# 2026.5.30-3 Benchmark 与 Metrics 打点系统

## TL;DR

- 已补第一阶段 metrics 系统：`MetricsRegistry` + `ScopedMetricsTimer`，支持 latency summary、counter、文本导出。
- `bench_client` 只保留终端输出和 `--output_json=path`，不再输出 metrics artifact。
- storage 进程不再支持 `metrics_output` / `metrics_dump_interval_ms` 文件导出；服务端 metrics 后续应走 HTTP `/metrics` 暴露。
- SDK route cache 之前只有配置，没有真正使用；曾实现并验证过缓存效果，但当前代码已按学习需要撤回该优化，仅保留 benchmark 与打点。
- PUT 性能差的主要原因不是本地 WAL fsync，而是 leader 同步 `flush_messages()` 给 follower 的 Raft RPC。
- GET 性能差的主要原因是每次线性一致读都会做 ReadIndex quorum heartbeat；曾验证过 quorum early-return 的优化收益，但当前代码已撤回该优化，仅保留原始 ReadIndex 路径与打点。
- 当前核心 metrics 不负责文件 IO；HTTP `/metrics` 和 `/debug/metrics` 适合下一阶段复用同一个 `MetricsRegistry`。

大样本结果：

```text
put_1t_2000:

  avg_us=3806.57
  p95_us=5571
  p99_us=10420

get_1t_2000:
  success=2000

mixed_4t_4000:
  success=3999
  failure=1
  failure_rate=0.00025
  avg_us=4525.92
  p95_us=8875
  p99_us=13063
```

结论：

- 单线程 PUT / GET 大样本没有错误率上升。
- 4 线程 mixed 有 1 次失败，错误率是 `0.025%`，不是明显大面积错误，但不是 0，需要解释。

失败原因：

```text
storage-3:
replica put failed, table_id = 2, shard_id = 0,
key = bench-key-000000000597,
msg = this pyt is not yet commit
```

同一时刻后续日志显示这个 key 随后被 apply：

```text
map put ok, key = bench-key-000000000597
```

所以这个失败不是 RPC timeout，不是 SDM route cache 问题，也不是写丢了；它是当前 `Replica::put()` 语义导致的：

1. leader `propose()` 后先 `flush_messages()` 一轮；
2. 再 `apply_committed_entries()`；
3. 如果此时 `commit_index < new_commit_idx`，直接返回 `NOT_YET_COMMIT`；
4. 在并发 mixed 下，目标日志可能下一轮 heartbeat / 其他请求 flush 时马上提交并 apply，但当前请求已经被 benchmark 计为 failure。

这个问题不应该简单在 SDK 里自动重试隐藏。原因是 `NOT_YET_COMMIT` 是“结果未知”的写：没有 client request id / 去重表时，SDK 自动重试 PUT/DEL 可能和其他客户端对同 key 的后续写发生重排。短期更合理的修复是 storage 写路径在返回前做 bounded wait / bounded drive commit；中长期应该引入 request id，支持幂等重试。

本轮 leader 是 `storage-2`，关键 metrics：

```text
storage.put.handler count=220 avg_us=3546.345 p95_us=6176
replica.put.total   count=220 avg_us=3545.109 p95_us=6176
replica.flush_messages count=585 avg_us=2466.371 p95_us=6736
```

结论：

- PUT 的大头在 `replica.flush_messages()`，也就是同步给 follower 发 AppendEntries 并等待响应。

- `raft.propose` 也不是大头。

短期优化方向：

- 可以考虑让 `flush_messages()` 中对多个 follower 的 AppendEntries 并发发送。
- 写请求只要达到多数派即可返回，剩余 follower 可由后续 heartbeat / append 补齐。
- 但这个改动会影响 Raft 消息调度语义，不能在本次顺手改，需要单独配 raft cluster test 和 E2E 验证。

## 当前瓶颈判断

### P0

暂无新的正确性 P0。

本次 ReadIndex early-return 初版曾导致单副本测试失败，原因是单副本没有 follower message，但 leader 自己已经满足 quorum。已修复为循环前判断 `success_cnt >= limit`，并通过 `replica_test`。

### P1

#### 1. PUT 同步复制 RPC 是当前主要瓶颈

证据：

```text
storage.put.handler avg_us=3546.345
replica.put.total avg_us=3545.109
replica.flush_messages avg_us=2466.371
persist.append_wal.fsync avg_us=38.462
raft.propose avg_us=77.541
```

影响：

- 单线程 PUT QPS 被同步 RPC 等待直接限制在几百级。
- follower 慢或网络抖动时，tail latency 会被放大。

建议：

- 短期：把 AppendEntries RPC 并发发送，并在多数派确认后返回。
- 中期：把 `flush_messages()` 改成异步 Raft transport，leader 写请求等待 commit future，不直接承担所有 follower RPC。
- 长期：引入 batch/group commit，合并多条 propose / append / fsync。

验证：

- `raft_cluster_test` 补“多数派成功即可提交，落后 follower 后续追赶”。
- E2E 补 follower 停止、恢复后的 log catchup。
- benchmark 比较 PUT avg/p95/p99。

#### 2. SDK route cache 之前没有生效

证据：

```text
关闭 cache: sdk.route_cache.disabled=200, sdk.route_cache.miss=200
开启 cache: sdk.route_cache.hit=100, sdk.route_cache.miss=100
```

影响：

- 每个 key 首次访问或禁用 cache 时都会查 SDM。
- 在高 QPS 或多 client 下会额外压 SDM。

当前修复：

- 已实现 SDK route cache。

后续建议：

- 当前容量满策略是 clear all，后续可以改 LRU。
- 当前 cache 粒度是 key，后续可考虑缓存 shard 路由范围，减少 key_count 很大时的首次 miss。

### P2

#### 1. MetricsRegistry 当前是 mutex + vector

影响：

- 高并发 benchmark 下，metrics 本身会有锁竞争和内存增长。
- 当前适合第一阶段定位，不适合作为最终高性能观测组件。

建议：

- 下一阶段改成 histogram bucket + atomic counter，避免每次记录都 push vector。
- 增加 Prometheus text exporter。
- 增加 `/metrics` 和 `/debug/metrics`。

#### 2. Benchmark LatencyRecorder 仍是全局 mutex

影响：

- 多线程 benchmark 下 client 侧 latency recorder 会引入额外竞争。

建议：

- 改成 per-thread recorder，结束后 merge。
- 多线程压测时再做这个，不阻塞当前单线程定位。

我做了什么

安装并启动了 Colima：

brew install colima
colima start --cpu 2 --memory 4 --disk 40 --runtime docker
当前 Docker context 已切到：

colima
绕过 Docker Hub 超时拉了镜像：

docker pull docker.1ms.run/grafana/grafana:11.2.0
docker tag docker.1ms.run/grafana/grafana:11.2.0 grafana/grafana:11.2.0

docker pull docker.1ms.run/prom/prometheus:v2.54.1
docker tag docker.1ms.run/prom/prometheus:v2.54.1 prom/prometheus:v2.54.1
然后在 metrics_demo 里启动成功：

docker compose up -d
