目前写完了这四个task

- CapacityChecker      这个就是用来检测replica目前是否有没有多或者少的，如果多了就去掉，如果少了就创建replica出来

- RoutingUpdater       这个是用来更新route的，只把shard上ready的replica更新到route上去，如果没有ready的话，虽然我们可以找到，但是route不会有。
                       然后还担当这对于ready的replicas里面选择出来leader

- HeartbeatCheckTask   这个是用来心跳check的，检查每一个资源池下面的node有没有挂掉，如果挂掉了，就把上面附着的replica设置成LOST，因为不是READY了，所以到时候
                       路由表上就没有他们了（只是说有这个流程，不是说由heartbeatcheck做）。

- ReplicaScheduler     这个是用来做replica调度的，对于我们目前shard上有问题的replica，我们得想办法把他们弄到shard上去，所以就得给他们分配node
                       所以这个就负责给replica分配node，分配node给replica后会更新他们的状态为ADDING。按照链路这里最后如果装好了，replica就会变成READY（只是说有这个流程，不是说由scheduler做），到时候就可以更新到路由表了

以上内容就会出现一些问题，首先是replica从LOST变成PENDING，需要怎么处理
第二，给replica分配到某个node上之后，replica的状态会变成ADDING，什么时候需要变成READY，我们如何判断？

这边关于第一个问题，目前的决定是，在CapacityChecker那边如果检测到了LOST的replica，就把他去掉，然后之后会补上个PENDING的replica。
同时这边意识到了一件事情，就是关于操作这边数据的并发性这边还没有考虑到，所以这边要上一下锁。

这边等到HeartBeatService实现了之后，开始完善数据的并发性。


// 4.17

- SdmStore ：唯一事实源，存 Table / Node / Replica / ResourcePool / ShardRoute
- TableService ：只负责接收 placement 类请求，写 Table
- NodeService ：只负责注册 node、维护资源池归属
- HeartBeatService ：吸收 storage 实际状态，并返回 desired replicas

- CapacityChecker ：只负责“数量对不对”
- ReplicaScheduler ：只负责“PENDING 分配到哪个 node”
- HeartBeatCheckTask ：只负责“node 是否超时、旧 replica 是否应删除/失效”
- RoutingUpdater ：只负责“从事实源重建路由 + leader 选择”
- RouteService ：只负责读路由，不做修复
