# 2026.5.30-2 E2E bug 修复与风险记录

## 本次实际修复

这次最终只保留 SDK 配置生命周期修复。

`KVClient` 构造函数里原本把构造入参 `conf` 直接传给 `SdmRouteClient` 和 `StorageClient`：

```cpp
KVClient::KVClient(const KVClientConf& conf)
    : conf_(conf), sdm_route_client_(conf), storage_client_(conf) {}
```

其中 `StorageClient` 内部保存的是 `const KVClientConf&`。在 E2E 的 `make_kv_client()` 中，`KVClientConf conf` 是函数内局部变量，`KVClient` 返回后，`StorageClient::conf_` 会变成悬空引用。结果是 `storage_timeout_ms`、log callback 等配置可能被读成未定义值，表现为重启或切主后 `Get` RPC 异常超时。

修复方式是让子 client 自己保存配置副本，这样构造时直接传 `conf` 也不会悬空：

```cpp
KVClient::KVClient(const KVClientConf& conf)
    : conf_(conf), sdm_route_client_(conf), storage_client_(conf) {}
```

子 client 内部保存配置值，而不是保存外部引用：

```cpp
KVClientConf conf_;
```

这样 `SdmRouteClient` 和 `StorageClient` 都不再依赖外部 `KVClientConf` 对象的生命周期。

## 已撤回但建议后续评估的风险点

### SDK RPC 错误重试策略

当前 `KVClient::should_invalidate_route()` 只在以下状态下刷新 route：

- `NOT_LEADER`
- `REPLICA_NOT_FOUND`
- `ROUTE_NOT_FOUND`

Storage gRPC 层面的 `Deadline Exceeded` / `Unavailable` 当前会被包装成普通 `ERROR`，不会触发 route refresh，也不会丢弃 cached stub。切主、重启、旧连接异常时，这可能导致客户端在短时间内持续打向坏连接或旧 leader。

建议后续单独设计 SDK 错误语义：

- gRPC transport error 使用 `RPC_ERROR`，不要和业务 `ERROR` 混在一起。
- `RPC_ERROR` 可以触发一次 route refresh。
- 对失败 endpoint 的 cached stub/channel 可以考虑驱逐或重建。
- 需要补充 SDK 层单测或 E2E，覆盖 storage 节点重启、连接断开、leader 切换后的 retry 行为。

### 内部 Raft RPC 没有 deadline

`RaftSender::send_request_vote()`、`send_append_entries()`、`send_install_snapshot()` 当前使用同步 gRPC 调用，但没有显式 deadline。

风险：

- follower 进程卡住但 TCP 连接未快速失败时，Raft tick 线程可能长时间阻塞。
- leader 的 heartbeat / replication / read-index quorum 路径会被拖慢。
- 在测试和真实环境中容易放大重启、kill、网络异常时的抖动。

建议后续评估：

- `RequestVote` / `AppendEntries` 设置较短 deadline。
- `InstallSnapshot` 设置更长但有限的 deadline。
- deadline 参数最好进入 storage 配置，而不是硬编码。

### AppendEntries response 可能高估 follower 复制进度

当前 `RaftNode::handle_append_response()` 在 `result.success` 时直接使用 leader 当前 `last_log_index_unlocked()` 更新 follower：

```cpp
match_index_[from] = last_log_index_unlocked();
next_index_[from] = last_log_index_unlocked() + 1;
```

风险：

- 如果本次 AppendEntries 只携带了较早的一批 entries，或者 leader 在 RPC 期间又追加了新日志，这里会把 follower 的复制进度推进到 leader 最新 index。
- 这可能造成 leader 错误判断多数派已经复制某些 entry，从而提前推进 commit index。

建议后续修复方向：

- 在发送 AppendEntries 时记录本次 RPC 实际覆盖到的 index。
- response 成功时只更新到本次实际 replicated index。
- raft cluster 单测需要覆盖 “RPC in-flight 期间 leader 追加新日志” 的场景。

### commit index 未变化时仍持锁刷盘

`try_update_commit_index()` 当前最后会无条件 `save_raft_meta()`。

风险：

- heartbeat、read-index 或重复 AppendEntries response 即使没有推进 `commit_index_`，也可能触发 `open/write/fsync/rename/fsync_dir`。
- 该函数在 Raft mutex 内执行，磁盘同步 IO 会扩大锁占用时间。
- 压力或慢盘场景下可能影响 write/read latency。

建议后续评估：

- 只有 `commit_index_` 真实推进时再持久化 raft meta。
- 或者进一步把持久化从 Raft mutex 临界区拆出去，但需要小心 crash consistency。

### Debug 日志可以保留为排障开关，而不是常开

排查期间曾在 Storage handler、Replica put/get、ReadIndex quorum 路径加过 debug 日志。最终已撤回，避免污染正常路径。

建议后续：

- 如果要保留类似日志，最好通过配置开关或 trace id 控制。
- 高频读写路径不要默认打过多日志。
- E2E 失败时可以临时打开 SDK attempt endpoint 日志，帮助判断请求是否到达 leader。

## 结论

本次 E2E 失败的直接根因是 SDK 配置悬空引用。其他 Raft / SDK retry 问题目前不作为本次 bug fix 提交，但它们是真实的稳定性风险，适合后续单独拆 PR/commit，并配套补测试。
